# KOVCHEG Blog — политика Module First

Этот документ обязателен для разработки KOVCHEG Blog 3.4+.

## Основное правило

Новая пользовательская функция сначала проектируется как модуль/плагин. В ядро добавляются только универсальные контракты, необходимые нескольким независимым функциям.

Примеры того, что должно находиться в ядре:

- загрузка и проверка модулей;
- события и хуки;
- маршрутизация;
- авторизация и права;
- реестр зон и виджетов;
- реестр типов контента;
- SEO collector;
- Embed Provider API;
- планировщик задач;
- миграционный API;
- безопасная работа с assets;
- логирование, кеш и диагностика;
- интерфейсы Studio для подключения разделов.

Примеры официальных модулей:

- поиск;
- комментарии;
- реакции;
- подписки;
- уведомления;
- портфолио;
- видео;
- музыка;
- галерея;
- формы;
- аналитика;
- карты;
- новостной портал;
- социальная лента;
- импорт RSS;
- интеграции с внешними сервисами.

## Почему это обязательно

- ядро остаётся небольшим и стабильным;
- функции можно отключать без удаления кода ядра;
- обновление ядра не перезаписывает расширения;
- разные сайты получают только нужные возможности;
- сторонние разработчики могут расширять систему;
- ошибка одного модуля не должна ломать весь сайт;
- функции можно выпускать и обновлять независимо;
- официальный Marketplace становится возможным без переписывания архитектуры.

## Проверка перед добавлением кода в ядро

Перед реализацией необходимо ответить:

1. Эта функция обязательна для каждого сайта?
2. Она является общим контрактом для других модулей?
3. Без неё невозможно безопасно загрузить или выполнить расширение?
4. Она относится к безопасности, маршрутизации, данным, правам или системному lifecycle?
5. Её невозможно отключить без нарушения целостности платформы?

Если большинство ответов отрицательные, функция реализуется модулем.

## Запрещённые практики

- изменение файлов ядра установщиком модуля;
- копирование логики модуля в layout темы;
- прямое добавление обязательного поиска, меню или социального блока в тему;
- изменение CSP строкой внутри стороннего шаблона;
- регистрация глобальных функций без namespace;
- выполнение SQL при каждом запросе вместо миграции;
- хранение секретов в manifest или публичных assets;
- скрытая передача данных внешнему сервису;
- удаление пользовательских данных при отключении модуля;
- fatal error при отсутствии необязательного модуля;
- зависимость одного модуля от внутреннего файла другого без публичного API;
- вставка произвольного JavaScript через обычный текстовый виджет.

## Lifecycle модуля

Модуль должен поддерживать состояния:

```text
not-installed
installed-disabled
enabled
update-available
incompatible
failed
quarantined
```

События lifecycle:

- validate;
- install;
- migrate;
- enable;
- boot;
- disable;
- update;
- uninstall;
- purge-data — только отдельным подтверждённым действием.

## Публичные API вместо внутренних файлов

Модуль использует только документированные интерфейсы:

- `Hooks`;
- `Router`;
- `WidgetRegistry`;
- `LayoutRegistry`;
- `EmbedRegistry`;
- `SeoCollector`;
- `PermissionRegistry`;
- `ContentTypeRegistry`;
- `StudioNavigationRegistry`;
- `CronRegistry`;
- `AssetRegistry`;
- `ModuleStorage`;
- `ModuleLogger`.

Внутренний класс, не объявленный стабильным API, может измениться без обратной совместимости.

## Версионирование API

- версия продукта и версия Module API разделяются;
- Module API меняется только осознанно;
- несовместимое изменение требует новой major-версии API;
- устаревшие методы получают deprecation-период;
- CI проверяет официальный набор контрактных тестов;
- модуль указывает поддерживаемый диапазон ядра;
- перед обновлением ядро проверяет совместимость активных модулей.

## Официальные и сторонние модули

### Официальный модуль

- хранится в организации проекта или подписан доверенным ключом;
- проходит полный CI;
- документирован;
- поддерживается вместе с релизом;
- может поставляться в стандартной сборке, но остаётся отключаемым.

### Сторонний модуль

- устанавливается только после предупреждения;
- получает минимально необходимые возможности;
- отображает автора, лицензию и источник;
- может быть помещён в quarantine;
- не считается частью гарантий ядра.

## Модули и темы

Тема не заменяет модуль.

Тема отвечает за:

- сетку;
- зоны;
- визуальные варианты;
- типографику;
- адаптивность;
- шаблоны отображения зарегистрированных сущностей.

Модуль отвечает за:

- данные;
- бизнес-логику;
- маршруты;
- виджеты;
- настройки;
- права;
- интеграции;
- миграции.

Portal Theme не должен содержать собственный жёстко вшитый поиск, комментарии или видео. Он только отображает соответствующие виджеты, если они размещены владельцем.

## Модули и SEO

Модуль не выводит `<title>`, canonical и глобальные Open Graph теги самостоятельно. Он передаёт структурированные данные через SEO Collector.

Модуль может добавить:

- Schema.org entity;
- sitemap provider;
- RSS provider;
- breadcrumb;
- Open Graph image;
- robots directive;
- canonical candidate;
- redirect provider.

Конфликты разрешает центральный SEO Manager.

## Модули и социальность

Социальные функции должны быть независимыми:

- комментарии можно отключить, сохранив реакции;
- реакции можно отключить, сохранив просмотры;
- подписки на автора могут работать без email-рассылки;
- уведомления могут получать события от разных модулей;
- модерация и антиспам подключаются через общий контракт;
- профиль автора не должен зависеть от конкретного шаблона.

## Модули и читаемые URL

Модуль регистрирует публичный route namespace и URL pattern. Ядро проверяет:

- конфликт со служебными путями;
- уникальность;
- безопасный slug;
- canonical;
- историю изменения адресов;
- редиректы;
- sitemap eligibility;
- правила доступа.

## Definition of Done для модуля

Модуль считается готовым, когда:

- устанавливается ZIP-пакетом;
- не изменяет ядро;
- имеет manifest;
- миграции идемпотентны;
- включается и отключается;
- ошибка обрабатывается без падения сайта;
- права зарегистрированы;
- маршруты покрыты тестами;
- виджеты покрыты тестами;
- данные не удаляются при отключении;
- README описывает установку и настройки;
- указаны автор, правообладатель и лицензия;
- проверена совместимость MySQL/MariaDB;
- отсутствуют секреты и runtime-данные;
- пройдены проверки безопасности.
