# KOVCHEG Layout & Widget Engine

Этот документ задаёт технический контракт визуального управления публичным сайтом. Он является обязательной основой для KOVCHEG Blog 3.4+ и Portal Theme.

## 1. Назначение

Layout Engine отвечает за зоны страницы и схемы расположения. Widget Engine отвечает за типы блоков и их экземпляры. Тема отвечает только за представление поддерживаемых зон, сетку и стили. Модуль может регистрировать собственные виджеты, зоны, настройки, assets и правила отображения.

Ни поиск, ни меню, ни профиль, ни список публикаций не должны быть жёстко вшиты в layout темы.

## 2. Термины

- **Zone** — именованная область шаблона, например `header.main` или `sidebar.right`.
- **Widget type** — зарегистрированный тип блока, например `core.menu` или `video.embed`.
- **Widget instance** — конкретный настроенный экземпляр виджета.
- **Placement** — размещение экземпляра в зоне с порядком и условиями видимости.
- **Layout** — схема колонок и зон для контекста страницы.
- **Context** — тип открытой страницы и её параметры: главная, статья, рубрика, автор, поиск и т. п.
- **Provider** — модульный обработчик внешнего содержимого, например YouTube или VK Видео.

## 3. Стандартные зоны

```text
header.top
header.main
header.bottom
page.before
layout.left
content.before
content.main
content.after
layout.right
page.after
footer.top
footer.columns
footer.bottom
```

Тема обязана объявлять:

- список поддерживаемых зон;
- допустимую ширину зоны;
- поддерживаемые варианты колонок;
- поведение зоны на мобильном устройстве;
- допускается ли sticky;
- поддерживаются ли вложенные контейнеры;
- ограничения для полноширинных виджетов.

Модуль может зарегистрировать дополнительную зону, но не может требовать изменения файлов темы. Если тема не поддерживает дополнительную зону, виджет остаётся выключенным и Studio показывает понятное предупреждение.

## 5. Реестр виджетов

Ядро предоставляет API регистрации:

```php
WidgetRegistry::register('core.menu', [
    'name' => 'Меню',
    'module' => 'core-navigation',
    'renderer' => MenuWidget::class,
    'settings' => MenuWidgetSettings::class,
    'cache' => 'context',
    'allowed_zones' => ['header.*', 'layout.*', 'footer.*'],
    'multiple' => true,
    'supports' => ['title', 'visibility', 'style'],
]);
```

Обязательные свойства типа:

- уникальный slug;
- владелец типа: ядро или модуль;
- название и описание;
- renderer;
- схема настроек;
- разрешённые зоны;
- возможность нескольких экземпляров;
- политика кеширования;
- требуемое право;
- список assets;
- поведение при отключении модуля.

## 6. Контракт renderer

Renderer получает только подготовленный контекст:

```php
interface WidgetRenderer
{
    public function render(
        WidgetInstance $instance,
        WidgetContext $context
    ): WidgetResult;
}
```

`WidgetContext` содержит:

- текущий маршрут;
- тип страницы;
- объект материала при наличии;
- пользователя и роль;
- тему и layout;
- язык;
- устройство/viewport-class без пользовательского fingerprinting;
- безопасный URL builder;
- CSP/asset collector.

Renderer не должен самостоятельно изменять глобальный layout, выполнять миграции или записывать настройки.

## 7. Studio: редактор виджетов

Интерфейс должен содержать:

- палитру доступных виджетов;
- зоны активного layout;
- drag-and-drop;
- клавиатурное перемещение как доступную альтернативу;
- включение/отключение;
- дублирование;
- редактирование настроек в боковой панели;
- предпросмотр desktop/tablet/mobile;
- публикацию черновика схемы;
- ревизии и откат;
- предупреждение об отключённом модуле;
- удаление placement отдельно от удаления instance.

Сохранение выполняется транзакционно. Нельзя получить частично сохранённую схему.

## 8. Условия видимости

`visibility_json` может задавать:

- тип страницы;
- конкретный маршрут;
- рубрику или тег;
- тип материала;
- автора;
- роль или статус входа;
- язык;
- desktop/tablet/mobile;
- дату начала и окончания;
- отрицательные условия.

Условия должны компилироваться в безопасную системную модель. Пользователь не вводит PHP или SQL.

## 9. Меню как виджет

Тип `core.menu` получает `menu_id` и параметры представления.

Настройки экземпляра:

- выбранное меню;
- горизонтальное/вертикальное представление;
- максимальная глубина;
- показывать иконки;
- показывать описание;
- мобильное сворачивание;
- заголовок;
- CSS-вариант темы.

Одно меню может использоваться:

- в `header.main`;
- в `layout.left`;
- в `layout.right`;
- в `footer.columns`;
- одновременно в нескольких местах.

## 10. Поиск как отключаемый модульный виджет

Поиск не является обязательной частью шапки.

- виджет можно удалить из всех зон;
- можно поставить в шапку, колонку, содержимое или подвал;
- поисковый backend регистрируется модулем;
- базовый поиск работает по публичным материалам;
- расширенные движки подключаются отдельными модулями;
- отключение поиска убирает публичную форму и маршрут либо переводит маршрут в корректный 404/410 согласно настройке.

## 11. Embed Provider API

Модули регистрируют провайдеры:

```php
EmbedRegistry::register('youtube', [
    'hosts' => ['youtube.com', 'www.youtube.com', 'youtu.be'],
    'resolver' => YouTubeProvider::class,
    'csp' => [
        'frame-src' => ['https://www.youtube-nocookie.com'],
        'img-src' => ['https://i.ytimg.com'],
    ],
]);
```

Provider обязан:

- проверить URL;
- извлечь безопасный идентификатор;
- вернуть нормализованные данные;
- создать iframe через системный renderer;
- определить aspect ratio;
- задать `sandbox`, `allow`, `loading` и `referrerpolicy`;
- вернуть fallback-ссылку;
- не принимать произвольный HTML от внешнего ответа без очистки.

Приоритет: YouTube, VK Видео, Rutube, Vimeo. Новые провайдеры добавляются модулями.

## 12. HTML и пользовательский код

Должны существовать два разных типа:

### Safe Embed

Для обычного редактора. Принимает URL, использует зарегистрированный Provider API и не разрешает произвольный HTML.

### Trusted HTML

Только для владельца или отдельного права `site.trusted_html`.

- HTML проходит allowlist sanitizer;
- `<script>` запрещён по умолчанию;
- inline event handlers запрещены;
- iframe разрешён только для разрешённых доменов;
- изменения журналируются;
- блок имеет визуальное предупреждение;
- CSP не ослабляется автоматически.

JavaScript-интеграции устанавливаются модулем, а не вставкой случайного `<script>` в виджет.

## 13. Кеширование

Политики:

- `none`;
- `request`;
- `public`;
- `context`;
- `user`.

Ключ кеша должен учитывать тип виджета, настройки, layout, язык, права и контекст. Персональные виджеты нельзя сохранять в публичном кеше.

Кеш сбрасывается при:

- изменении экземпляра;
- изменении placement;
- публикации материала, от которого зависит виджет;
- включении/отключении модуля;
- смене темы;
- изменении меню.

## 14. Поведение при ошибках

Ошибка одного виджета не должна обрушать страницу.

- публичный сайт скрывает сломанный виджет и пишет ошибку в журнал;
- в debug-режиме показывается диагностическая рамка;
- Studio показывает состояние виджета;
- критический виджет может объявить явную политику отказа;
- отключённый модуль не вызывает fatal error;
- placement сохраняется, чтобы восстановиться после включения модуля.

## 16. SEO и URL

Widget Engine не должен генерировать второй canonical или конфликтующие метатеги. SEO собирается централизованным сервисом.

Виджет или модуль может передать:

- Schema.org entity;
- Open Graph media;
- breadcrumb item;
- preload/preconnect resource;
- noindex-запрос для служебной страницы.

Окончательное решение принимает SEO Manager.

Официальные модули могут поставлять:

- последние комментарии;

- подписку;
- реакции;
- избранное;

Каждая социальная функция должна отключаться независимо.
